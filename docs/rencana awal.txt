Berikut pendekatan yang lebih “production-grade” untuk SaaS **platform ujian sekolah** yang Anda rancang. Saya fokus ke *arsitektur, tech stack, dan setup awal* — bukan coding.

---

# 🧠 1. Karakter Sistem (Requirement Engineering)

Dari deskripsi Anda, ini bukan sekadar web app biasa. Ini adalah:

* **Multi-tenant SaaS** (banyak sekolah dalam 1 platform)
* **High concurrency system** (ujian serentak ribuan siswa)
* **Hybrid client system** (web + proktor app + secure browser)
* **Role-based system kompleks**
* **Event-based system** (ujian berbasis sesi & waktu)

👉 Ini implikasinya:

* Harus scalable (horizontal scaling)
* Harus low latency (real-time monitoring)
* Harus secure (anti-cheat, anti-exploit)
* Harus fault-tolerant (ujian tidak boleh crash)

---

# 🏗️ 2. Arsitektur Sistem (Disarankan)

## 🔹 High-Level Architecture

```
[ Client Browser (Siswa) ]
        |

        |

   |        |

   |        |
[ DB ]    [ Redis / PubSub ]
   |
[ Analytics / Warehouse ]
```

## 🔹 Komponen Penting

### 1. **Core Backend (Business Logic)**

* Manajemen user, role, soal, ujian
* Multi-tenant isolation
* Scoring engine

### 2. **Realtime Engine**

* Monitoring ujian (proktor)
* Status siswa (online/offline)
* Timer ujian sinkron

### 3. **Exam Engine (Critical)**

* Randomisasi soal
* Anti refresh exploit
* Auto save jawaban
* Time locking

### 4. **Client Layer**

* Secure browser (lock environment)
* Proktor dashboard (monitor live)

## 🔐 Fitur Wajib

### 1. Lockdown System

* Disable:

  * Alt+Tab
  * Ctrl+C/V
  * Right click
  * DevTools

```js
globalShortcut.register('Alt+Tab', () => false)
```

---

### 2. Fullscreen Enforcement

```js
win.setFullScreen(true)
win.setKiosk(true)
```

---

### 3. Token-based Access

Flow:

```
Login → Server generate token → inject ke browser
```

---

### 4. Heartbeat System

Client kirim:

```
POST /heartbeat
```

Jika tidak aktif:
→ auto submit / flag cheating

---

### 5. Anti Multiple Device

* 1 user = 1 session
* Redis lock

---

### 6. Screen Monitoring (Opsional advanced)
