# 🚨 Makassar Ujian: Incident Response Runbook

This runbook outlines the Standard Operating Procedures (SOP) for resolving critical alerts generated by the Prometheus/Alertmanager monitoring stack. 

**DO NOT PANIC.** Follow the steps below systematically to restore service stability.

---

## 🛑 1. Incident: High Latency (p95 > 2s)
**Alert Name:** `HighHttpLatency`  
**Severity:** WARNING  
**Responsible:** Backend Engineer / SRE  

### Symptoms:
- Users complain about the application being slow.
- The `app_latency_p95:5m` metric in Grafana is > 2 seconds.

### Immediate Actions:
1. **Identify the Route:** Check the alert details in Telegram or Alertmanager to find the exact `$labels.route` causing the spike.
2. **Check Distributed Traces:**
   - Open **Grafana -> Explore -> Tempo**.
   - Query traces for the specific route. Look at the trace waterfall to see if the delay is in the DB (Slow Query), Redis, or an external API.
3. **Database Check:** If the DB is the bottleneck, proceed to the "DB Slow" runbook below.
4. **Temporary Mitigation:** If a specific heavy endpoint is under DDoS, apply strict rate limiting in `RouteServiceProvider` or Nginx.

---

## 🚨 2. Incident: Redis Saturation
**Alert Name:** `RedisSaturation`  
**Severity:** CRITICAL  
**Responsible:** SRE / DevOps  

### Symptoms:
- Redis memory is > 80% full.
- Operations > 5000 ops/sec.
- Application throws "Connection refused" or Redis timeout errors.

### Immediate Actions:
1. **Restart Redis (Temporary Relief):** 
   ```bash
   docker restart makassarujian-redis-1
   # Or using Sail:
   ./vendor/bin/sail restart redis
   ```
2. **Flush Non-Critical Cache (If Memory is Full):**
   ```bash
   ./vendor/bin/sail artisan cache:clear
   ```
3. **Scale Capacity (Permanent Fix):** 
   - If using cloud (AWS ElastiCache), trigger an immediate scale-up to a larger node type or add Read Replicas.
4. **Investigate Root Cause:** Check if the Autosave feature is hammering Redis without debouncing from the frontend.

---

## 🚨 3. Incident: Queue Stuck
**Alert Name:** `QueueStuck`  
**Severity:** CRITICAL  
**Responsible:** Backend Engineer  

### Symptoms:
- Background jobs (like score calculation or email sending) are not processing.
- Queue size > 10,000 but throughput is < 10 jobs/min.

### Immediate Actions:
1. **Restart Horizon Workers:** The workers might be deadlocked or out of memory.
   ```bash
   ./vendor/bin/sail artisan horizon:terminate
   ```
   *(Supervisor will automatically restart them).*
2. **Check Deadlocks:** Check PostgreSQL logs for `deadlock detected`. If database locks are preventing jobs from completing, kill the blocking queries.
3. **Scale Workers:** If the queue is moving but just too slow, increase the `processes` count in `config/horizon.php` and deploy.

## 🚨 4. Incident: Database Slow / Overloaded
**Alert Name:** *(Various latency / query alerts)*  
**Severity:** CRITICAL  
**Responsible:** Database Administrator (DBA) / SRE  

### Symptoms:
- CPU usage on the PostgreSQL server is near 100%.
- All application requests begin timing out.

### Immediate Actions:
1. **Kill Slow Queries:**
   Access the DB console and kill queries taking longer than 1 minute:
   ```sql
   SELECT pg_terminate_backend(pid) 
   FROM pg_stat_activity 
   WHERE state = 'active' AND (now() - query_start) > interval '1 minute';
   ```
2. **Enable Read Replica (Failover):** 
   If the primary DB is overwhelmed by read queries (e.g., loading exam statistics), switch the `DB_READ_HOST` in the `.env` file to point to the Standby Database.
   ```bash
   # Run the custom failover script
   php artisan dr:failover-region
   ```
3. **Analyze Traces:** Use Tempo to find out which query is causing the spike. Missing indexes are the usual culprit.

---

## 🚨 5. Incident: Autosave Failure
**Alert Name:** `AutosaveFailure`  
**Severity:** CRITICAL  
**Responsible:** Backend Engineer  

### Symptoms:
- 5xx error rate spiking specifically on `/api/exam/save-answer`.
- Students report their answers aren't saving.

### Immediate Actions:
1. **Check Disk Space:** Check if the DB or Redis has run out of disk space (`df -h`).
2. **Check Logs:** Open Grafana **Loki** and filter for `{route="api/exam/save-answer", status=~"5.."}` to read the exact Exception message.
3. **Database Write Check:** Ensure the Primary Database is not in "Read Only" mode or suffering from connection exhaustion. 
4. **Emergency Mitigation:** If DB is down, route Autosave payloads directly into a Redis Queue as a buffer until the DB recovers.
